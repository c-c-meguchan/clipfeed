# なぜ「通常 Run」はラグるのに「Profile ビルド」はサクサクなのか

## 原因：Debug と Release の違い

- **通常の Run（▶）** … **Debug** 構成でビルドされる  
  - `SWIFT_OPTIMIZATION_LEVEL = "-Onone"`（最適化なし）  
  - SwiftUI の body や Combine の処理がほぼ最適化されず、重くなりやすい  
  - その結果、再コピー時のハイライト／トーストが遅れて感じる

- **Profile（Instruments で計測）** … **Release** 構成でビルドされる  
  - 最適化が有効（-O など）  
  - 同じコードでも体感がかなり軽くなる  
  - データやロジックは同じなので、ペーストなどの動作はどちらも同じ

つまり **「レコード用ビルドだけサクサク」なのは、そのビルドが Release だから**で、  
データが入っていない／入っているの差ではありません。

---

## 普段からサクサク使いたい場合

### 方法 A：Run でも Release で動かす（開発中に体感を確認したいとき）

1. Xcode で **Product → Scheme → Edit Scheme…**（⌘<）
2. 左の **Run** を選択
3. **Info** タブの **Build Configuration** を **Debug** から **Release** に変更
4. **Close**

これで **Run（▶）** でも Release ビルドが使われ、Profile のときと同じように軽くなります。

- **注意**: Release ではデバッグがしづらくなります（ブレークポイントや `print` の見え方など）。  
  開発中は Debug のままにして、**「体感確認用」のときだけ Run を Release にする**運用がおすすめです。

### 方法 B：配布用アプリ（Archive）はそのまま

**Product → Archive** で作るアプリはもともと **Release** でビルドされるので、  
ユーザーがインストールして使うときは、すでにサクサクな方のビルドになっています。

---

## まとめ

| 操作           | ビルド構成 | 体感     |
|----------------|------------|----------|
| 通常の Run（▶） | Debug      | ラグりやすい |
| Profile        | Release    | サクサク   |
| Archive        | Release    | サクサク   |

「レコード用ビルドだけでサクサク」「データは入っている」というのは、  
**Release で動いているかどうかの差**で説明できます。
